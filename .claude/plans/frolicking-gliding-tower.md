# Plan: Changesets + Dagger CI

## Context

The monorepo has no automated versioning/publishing and uses a basic GitHub Actions CI with Bun. We're replacing the CI entirely with Dagger (TypeScript SDK) and adding Changesets for version management and npm publishing.

**Note:** `.github/` is in `.gitignore` (though `ci.yml` is already tracked). New workflow files will need `git add -f`.

---

## Files to create

### 1. `.changeset/config.json` — Changesets configuration
```json
{
  "$schema": "https://unpkg.com/@changesets/config@3.1.1/schema.json",
  "changelog": "@changesets/cli/changelog",
  "commit": false,
  "fixed": [],
  "linked": [],
  "access": "public",
  "baseBranch": "main",
  "updateInternalDependencies": "patch",
  "ignore": []
}
```
- `access: "public"` — required for scoped `@semi-solid/*` packages
- `updateInternalDependencies: "patch"` — bumps `@semi-solid/solid` when its dep `@semi-solid/runtime` changes

### 2. `.changeset/README.md` — Standard boilerplate (auto-generated by `pnpm changeset init`)

### 3. Dagger module files

**`dagger.json`** (repo root) — module manifest pointing to `.dagger/` source

**`.dagger/package.json`**, **`.dagger/tsconfig.json`** — Dagger module config

**`.dagger/src/index.ts`** — The pipeline module with these functions:

| Function | Trigger | Description |
|----------|---------|-------------|
| `baseEnv(source)` | (internal) | Node 22 + pnpm + cached store + deps installed |
| `test(source)` | Every push/PR | Runs `pnpm run test` |
| `buildPackages(source)` | Every push/PR | Runs `pnpm run build:packages` (tsup) |
| `buildAllThemes(source)` | Every push/PR | Builds all 4 brand/locale combos sequentially |
| `deployTheme(source, ...)` | Main only | Builds + deploys one theme via Shopify CLI |
| `deployStagingAll(source, ...)` | Main only | Orchestrates all 4 deploys |

Uses `dag.cacheVolume("pnpm-store")` for dependency caching. Base image: `node:22-slim` (may need Playwright image if storybook tests require it — will iterate).

### 4. `.github/workflows/ci.yml` — Replace existing with Dagger-based workflow

```yaml
name: CI
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:          # dagger call test --source=.
  build:         # dagger call build-packages --source=.  (needs: test)
  build-themes:  # dagger call build-all-themes --source=. (needs: test)
  deploy-staging: # dagger call deploy-staging-all --source=. ... (main only, needs: build + build-themes)
```

All jobs use `dagger/dagger-for-github@v8` action. Secrets passed via `env:SECRET_NAME` protocol.

### 5. `.github/workflows/release.yml` — Changesets publish workflow

Uses `changesets/action@v1` directly (not Dagger — the git operations for creating version PRs, committing, and tagging are inherently GitHub-specific and don't benefit from containerization).

```yaml
name: Release
on:
  push:
    branches: [main]
concurrency: ${{ github.workflow }}-${{ github.ref }}

jobs:
  release:
    # changesets/action creates a "Version Packages" PR when changesets exist
    # When that PR merges, it publishes to npm
    steps:
      - pnpm install
      - changesets/action with version + publish scripts
    env: GITHUB_TOKEN, NPM_TOKEN
```

### 6. `.npmrc` — NPM auth for CI publishing
```
//registry.npmjs.org/:_authToken=${NPM_TOKEN}
```

## Implementation order

1. Install changesets: `pnpm add -Dw @changesets/cli` + `pnpm changeset init` + edit config
2. Add release scripts to root `package.json`
3. Create `.npmrc`
4. Initialize Dagger: `dagger init --sdk=typescript --name=semi-solid --source=.dagger`
5. Write `.dagger/src/index.ts` pipeline
6. Replace `.github/workflows/ci.yml`
7. Create `.github/workflows/release.yml` (force-add to git since `.github/` is gitignored)
8. Update `.gitignore` with Dagger exclusions

## Verification

1. `dagger call test --source=.` — runs tests locally in container
2. `dagger call build-packages --source=.` — builds packages in container
3. Push a branch — CI should run test + build jobs via Dagger
4. `pnpm changeset` — creates a changeset file interactively
5. Merge to main — release workflow creates "Version Packages" PR
